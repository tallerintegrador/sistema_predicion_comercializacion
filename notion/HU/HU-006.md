# HU-006 — Endpoint de reposición de compras

> **Como** sistema cliente, **quiero** un endpoint HTTP que, dado mi histórico y mis parámetros
> logísticos, me devuelva cuánto reponer y cuándo, **para** integrarlo sin reimplementar la lógica.

- **Capa:** API — Fase 3. **Endpoint:** `POST /purchases`.
- **Código:** `src/spc/api/routers/compras.py`, `src/spc/api/schemas/compras.py`,
  `src/spc/service/compras_service.py` (lógica de HU-002).
- **Motor:** reutiliza `regresion_v3` vía VENTAS (sin modelo propio). **Tests:** `tests/api/test_compras_api.py`.

---

## 1. Objetivo y alcance

Exponer el campo **COMPRAS** del contrato. El cliente envía el mismo bloque `history` (de donde se
calcula internamente el pronóstico de demanda) y los `replenishment_params` logísticos por producto
(`current_stock`, `lead_time_days`, `target_coverage_days`). La respuesta es la cantidad a reponer y
el punto de reorden. **No hay modelo propio**: es lógica de negocio (ver **HU-002**).

## 2. Qué se hizo

### 2.1. Router fino (`routers/compras.py`)

`POST /purchases` con `response_model=ComprasResponse`, ejemplos en Swagger y errores documentados
(400 producto sin histórico / 422 entrada mal formada). Convierte Pydantic→dict y delega en
`compras_service.reponer`.

### 2.2. Esquemas estrictos (`schemas/compras.py`)

- `ParametroReposicion` con `extra="forbid"`: `current_stock ≥ 0`, `lead_time_days > 0`,
  `target_coverage_days > 0`. Listas con ≥ 1 elemento.
- `RecomendacionItem` con `expected_demand_horizon`, `reorder_point`, `replenishment_quantity` (≥ 0)
  y `justification` (la fórmula aplicada, para auditar).

### 2.3. Servicio: aritmética de inventario sobre el pronóstico (`compras_service.py`)

Reutiliza `ventas_service.forecast_diario` (un solo pronóstico hasta el horizonte máximo pedido) y
aplica la política de **días de cobertura** con `safety = 30 % × demanda(lead)` (detalle completo en
HU-002). Un producto sin histórico → `SolicitudInvalida` (**400**).

## 4. Resultados (contrato de la respuesta)

```json
{
  "field": "purchases",
  "recommendation": [
    {
      "store_id": "1",
      "product_id": "BEVERAGES",
      "expected_demand_horizon": 12200.0,
      "reorder_point": 5400.0,
      "replenishment_quantity": 11300.0,
      "justification": "forecast_demand(lead_time + coverage) + safety_stock - current_stock"
    }
  ],
  "metadata": {"assumption": "stock de seguridad = 30% de la demanda en lead time; …", "policy": "coverage_days"}
}
```

- Cobertura de tests (`tests/api/test_compras_api.py`): caso válido + inválidos (producto sin
  histórico → 400, parámetro fuera de rango → 422), **nunca un 500**.
- La recomendación es **trazable** (`justification`) y nunca negativa.

