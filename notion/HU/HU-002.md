# HU-002 — Reposición de compras

> **Como** responsable de compras, **quiero** saber cuánto reponer de cada producto y cuándo
> dispararlo, **para** evitar quiebres de stock sin sobre-comprar, partiendo del pronóstico de
> demanda.

- **Capa:** Servicio / lógica de negocio — Fase 3. **No tiene modelo propio.**
- **Código principal:** `src/spc/service/compras_service.py`.
- **Reutiliza:** el pronóstico diario de VENTAS (`HU-001` / `regresion_v3`).
- **Se expone en:** `POST /purchases` (ver `HU-006`). **Decisión:** `docs/decisiones/0007-capa-api-fase3.md` §5.

---

## 1. Objetivo y alcance

COMPRAS **no entrena nada**: es **lógica de inventario** que combina el pronóstico de demanda
(HU-001) con los parámetros logísticos que envía el cliente, y deriva por producto:

- la **demanda esperada** en la ventana (lead time + cobertura),
- el **punto de reorden**,
- la **cantidad a reponer**.

Mantener COMPRAS sin modelo propio es deliberado: la demanda ya la estima la regresión; añadir otro
modelo duplicaría la fuente de verdad. Aquí solo hay aritmética de inventario auditable.

## 2. Qué se hizo (política y fórmulas)

Política elegida (acordada con la validadora): **días de cobertura**. El stock de seguridad es un
porcentaje de la demanda durante el *lead time*.

Constante de política (no de artefacto): `FACTOR_STOCK_SEGURIDAD = 0.30`
(`compras_service.py:28`). Para cada producto:

```
demanda_lead       = Σ pronóstico_diario[0 : lead_time]
demanda_horizonte  = Σ pronóstico_diario[0 : lead_time + cobertura]
stock_seguridad    = 0.30 × demanda_lead
punto_reorden      = demanda_lead + stock_seguridad
cantidad_a_reponer = max(0, demanda_horizonte + stock_seguridad − stock_actual)
```

Pasos del servicio (`reponer`, `compras_service.py:31`):

1. **Valida** que haya parámetros y traduce el histórico al dataset analítico (`adaptador`).
2. **Rechaza** productos sin histórico → `SolicitudInvalida` (HTTP **400**): sin historia no se
   puede pronosticar su demanda.
3. Pronostica **una sola vez** hasta cubrir el horizonte más largo pedido
   (`forecast_diario`, reutilizado de VENTAS) y luego recorta por producto.
4. Aplica la aritmética de inventario y devuelve la respuesta del contrato, incluyendo un
   `metadata.assumption` que **expone el supuesto** del 30 % (transparencia para el cliente).

> **Por qué `current_stock`, `lead_time_days`, `target_coverage_days` los pone el cliente:** son
> datos de su operación (proveedor, política), no algo que SPC pueda inventar. SPC documenta el
> supuesto del colchón y deja que el cliente ajuste su política.

## 4. Resultados (ejemplo de salida)

Petición de ejemplo (`examples/api/compras_request.json`): histórico + `replenishment_params` con
`current_stock`, `lead_time_days`, `target_coverage_days` por producto. Respuesta:

```json
{
  "field": "purchases",
  "recommendation": [
    {
      "store_id": "1",
      "product_id": "BEVERAGES",
      "expected_demand_horizon": 12200.0,
      "reorder_point": 5400.0,
      "replenishment_quantity": 11300.0,
      "justification": "forecast_demand(lead_time + coverage) + safety_stock - current_stock"
    }
  ],
  "metadata": {
    "assumption": "stock de seguridad = 30% de la demanda en lead time; demanda y lead time aproximados; revisar política del cliente",
    "policy": "coverage_days"
  }
}
```

- Cada recomendación trae su **`justification`** (la fórmula aplicada): la decisión es **auditable**,
  no una caja negra.
- `replenishment_quantity` nunca es negativa (se recorta a 0 si el stock ya cubre el horizonte).

### Mejora diferida (documentada)

Política por **nivel de servicio** (z-score sobre la variabilidad de la demanda) en lugar de
% fijo; queda documentada en el ADR 0007 y no se implementó en este cierre.
