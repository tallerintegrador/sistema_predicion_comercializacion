# HU-005 — Endpoint de pronóstico de ventas

> **Como** sistema cliente, **quiero** un endpoint HTTP que reciba mi histórico y me devuelva el
> pronóstico de demanda, **para** integrarlo sin conocer el modelo interno.

- **Capa:** API — Fase 3. **Endpoint:** `POST /sales`.
- **Código:** `src/spc/api/routers/ventas.py`, `src/spc/api/schemas/ventas.py`,
  `src/spc/service/ventas_service.py`.
- **Motor:** artefacto `regresion_v3` (HU-001). **Tests:** `tests/api/test_ventas_api.py`.
- **Decisión:** `docs/decisiones/0007-capa-api-fase3.md`.

---

## 1. Objetivo y alcance

Exponer el campo **VENTAS** del contrato de datos por HTTP. El cliente envía `granularity`,
`horizon` y el bloque `history` (con **nombres genéricos**: `store_id`, `product_id`, `units_sold`,
`on_promotion?`, `transactions?`, `event_active?`) y recibe, por `(date, store_id, product_id)`, la
`forecast_demand` en **unidades**. La separación de capas es estricta: el **router** habla HTTP, el
**servicio** traduce el contrato y orquesta, el **motor** solo predice.

## 2. Qué se hizo

### 2.1. Router fino (`routers/ventas.py`)

`POST /sales` con `response_model=VentasResponse`, ejemplos en Swagger y respuestas de error
documentadas (400/422). Convierte Pydantic→dict y delega en el servicio; **no contiene lógica de
negocio**.

### 2.2. Esquemas Pydantic estrictos (`schemas/ventas.py` + `schemas/comunes.py`)

- `extra="forbid"`: un campo mal escrito (`horizont`) se **rechaza** (no se ignora).
- Rangos del contrato: `horizon > 0` (≤ 365), `units_sold ≥ 0`, `forecast_demand ≥ 0`, `history` con
  ≥ 1 elemento. `store_id`/`product_id` aceptan `str`/`int` y se normalizan a `str`.
- `interval_80` queda como **campo opcional documentado pero no emitido** (los intervalos se
  difirieron en Fase 2); los routers usan `response_model_exclude_none=True`.

### 2.3. Servicio: orquesta el pronóstico recursivo (`ventas_service.py`)

1. `adaptador.historico_a_analitico(history)` traduce contrato → dataset analítico.
2. `forecast_diario` añade el **esqueleto futuro** y llama al **pronóstico recursivo**
   `pronosticar_horizonte` del artefacto (autorregresivo, como en producción).
3. `granularity` `week`/`month` **agrega (suma)** el pronóstico diario; la `date` de salida es el
   inicio del período.
4. `model` y `metadata` (escala, transformación interna) se leen del **meta del artefacto**, nunca
   se hard-codean → si mañana sale `regresion_v4`, la respuesta lo refleja sin tocar código.

### 2.4. Carga de artefactos una sola vez

Se cargan en el arranque (lifespan, `main.py:74`) y se inyectan por `Depends(obtener_registro)`; los
routers **no cargan por petición**. La versión se resuelve por **glob** (`regresion_v*.joblib` → la
mayor).

## 4. Resultados (contrato de la respuesta)

```json
{
  "field": "sales",
  "model": "regresion_v3",
  "forecast": [
    {"date": "2017-08-03", "store_id": "1", "product_id": "BEVERAGES", "forecast_demand": 1742.5},
    {"date": "2017-08-04", "store_id": "1", "product_id": "BEVERAGES", "forecast_demand": 1690.2}
  ],
  "metadata": {"scale": "units", "internal_transform": "log1p"}
}
```

- `model` = versión real del artefacto (leída del meta).
- Cobertura de tests (`tests/api/test_ventas_api.py`): caso válido (forma == ejemplo del contrato) y
  casos inválidos (campo faltante, rango inválido, producto sin histórico) → siempre error controlado
  (422/400), **nunca un 500**.

> **Nota de IDs desconocidos:** una tienda/producto nuevo **no es error**; degrada con elegancia (la
> categórica desconocida cae a `NaN` y el pronóstico se sostiene sobre rezagos + calendario). El error
> de negocio (400) ocurre solo si el producto **no está en el histórico** enviado.
