# HU-008 — Interfaz para cargar datos y ver predicciones

> **Como** usuario de la demo, **quiero** ingresar o cargar datos y ver las predicciones,
> **para** visualizar el valor del producto.

- **Capa:** Frontend web (React + Vite + TypeScript). **Sprint 2: Semana 12.**
- **Estado:** ✅ Implementada (local). Consume los tres dominios + la predicción auto-entrenada.
- **Código clave:** `frontend/src/pages/` (Sales/Purchases/Inventory/Libre), `frontend/src/api/`
  (`client.ts`, `endpoints.ts`, `types.ts`), `frontend/src/hooks/usePrediction.ts`.
- **Decisiones:** ADR-0012 (frontend web), 0015 (rediseño experiencia), 0021/0022 (entrada/filtros),
  0023 (predicción agnóstica auto-entrenada).

---

## 1. Objetivo y alcance

Dar una **interfaz web** donde un usuario no técnico pueda **traer sus datos** y obtener
predicciones de los tres dominios (Ventas, Compras, Almacén) **sin tocar el motor ni la API a mano**.
La UI **nunca fabrica datos**: cuando una capacidad del backend no existe, el control se muestra
deshabilitado con la etiqueta «Próximamente» (ver `docs/alineacion_frontend_backend.md`).

## 2. Qué se hizo

### 2.1. Varias formas de **ingresar datos**

- **Manual / asistido** — el flujo guiado (`components/prediccion/PrediccionGuiada.tsx`) arma la
  petición campo por campo.
- **Excel** — plantilla descargable y carga de archivo (`api/endpoints.ts`: `downloadTemplate`,
  `uploadExcel` / `uploadAutoExcel`); el backend la valida contra el contrato.
- **Demo cargada** — datos sintéticos ricos listos para usar (`demo/retailDemo.ts`) para mostrar el
  producto sin que el usuario traiga nada (ver **HU-010**).

### 2.2. Una página por dominio que **consume la API**

- `pages/SalesPage.tsx`, `PurchasesPage.tsx`, `InventoryPage.tsx` y `LibrePage.tsx` (modo agnóstico).
- Llaman al cliente tipado `api/endpoints.ts` (`postSales`, `postPurchases`, `postInventory`, y los
  `postAuto*` de ADR-0023), que envuelve `fetch` y traduce el **error uniforme** del backend
  (`{error:{type,message,details}}`) a una excepción tipada `ApiError`.

### 2.3. Estado de la predicción manejado en un hook

- `hooks/usePrediction.ts` orquesta la llamada y distingue **resultado en línea (200)** de
  **comprobante de lote (202 `JobAccepted`)**, haciendo *polling* del job cuando aplica.
- Errores y validaciones se muestran en `components/ErrorPanel.tsx`; el origen de datos en
  `components/DataSourcePanel.tsx` y la previsualización del histórico en `HistoryPreview.tsx`.

### 2.4. Detrás de autenticación (ADR-0014/0015)

- Login + onboarding + sidebar filtrado por rol (`pages/LoginPage.tsx`, `OnboardingPage.tsx`,
  `auth/AuthContext.tsx`). El token de sesión viaja en cada llamada (`api/client.ts`).

## 3. Código puntual

| Qué | Dónde |
|-----|-------|
| Páginas por dominio (entrada + resultado) | `frontend/src/pages/SalesPage.tsx`, `PurchasesPage.tsx`, `InventoryPage.tsx`, `LibrePage.tsx` |

| Carga de datos (manual / Excel / demo) | `frontend/src/components/data/`, `components/prediccion/PrediccionGuiada.tsx`, `demo/retailDemo.ts` |
| Conciliación UI ↔ backend | `docs/alineacion_frontend_backend.md` |

## 4. Resultados (criterio de "hecho")

- ✅ **Consume los endpoints de la API** (los tres dominios + auto-entrenado).
- ✅ **Permite ingresar datos** (manual, Excel con plantilla, demo) y **muestra resultados** de los
  tres dominios.
- ✅ Maneja **errores del contrato** con mensajes claros (sin pantallas en blanco ni 500 crudos).
- ✅ Soporta el **modo lote** (202 → *polling*) además del modo en línea.

## 5. Cómo probar

```powershell
# Backend
venv\Scripts\python -m uvicorn spc.api.main:app
# Frontend (otra terminal)
cd frontend; npm install; npm run dev
# Abrir la URL que imprime Vite (p. ej. http://localhost:5173) → Login (256317 / 256317)
```
