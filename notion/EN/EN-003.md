# EN-003 — Contenerizar la aplicación con Docker

> **Habilitador técnico.** Empaquetar la API y sus artefactos en una imagen Docker para
> estandarizar el entorno y facilitar el despliegue reproducible.

- **Ámbito:** Infraestructura / despliegue. **Sprint 2: Semana 12.**
- **Estado:** 🟡 **Parcial.** `Dockerfile` de la API **listo y reproducible**;
  `docker-compose` (API + base de datos) **pendiente** (depende de **EN-002**).
- **Archivos clave:** `Dockerfile`, `.dockerignore`, `requirements-api.txt`.
- **Habilita:** **EN-004** (despliegue) y se apoya en la decisión de **carga sin reentrenar**
  (`docs/decisiones/0007-capa-api-fase3.md`).

---

## 1. Objetivo y alcance

Dejar la API **lista para correr en cualquier máquina** sin instalar Python ni dependencias a mano,
y horneando los **artefactos del motor** en la imagen para que el servicio **arranque, cargue y
prediga** sin volúmenes, sin descargas externas y **sin reentrenar** (ADR-0009).

La imagen está pensada para plataformas **PaaS tipo Render / Railway**: el puerto lo inyecta la
plataforma vía `$PORT` y la app hace *bind* en `0.0.0.0`.

## 2. Qué se hizo

### 2.1. Imagen de servicio mínima (`Dockerfile`)

- Base **`python:3.11-slim`** (etapa `runtime`): imagen chica, solo runtime (no entrena).
- **Variables de entorno** afinadas: `PYTHONUNBUFFERED` (logs en vivo en Render/Railway),
  `PYTHONDONTWRITEBYTECODE`, `PIP_NO_CACHE_DIR`, y `PYTHONPATH=/app/src` (layout `src/` → `import spc`
  funciona **sin instalar** el paquete).
- **`libgomp1`** instalado a propósito: es el runtime de OpenMP que **LightGBM y XGBoost** necesitan;
  sin él, el import del booster falla (`libgomp.so.1: cannot open shared object file`).

### 2.2. Capas cacheables (build rápido)

- Las **dependencias primero**: `COPY requirements-api.txt` + `pip install` en una capa propia.
  Mientras `requirements-api.txt` no cambie, Docker **reutiliza la capa** y el build es rápido.
- Luego el **código y los artefactos**: `COPY src/` y `COPY models/`. Los `.joblib` del motor se
  **hornean en la imagen** → el contenedor es autosuficiente.

### 2.3. Endurecimiento básico

- **Usuario sin privilegios** (`appuser`, uid 1000): no corre como root (buena práctica de seguridad).
- `EXPOSE 8000` documental; el arranque usa **shell form** para expandir `$PORT`:
  `uvicorn spc.api.main:app --host 0.0.0.0 --port ${PORT:-8000}`.
- **1 worker** deliberado: el motor carga ~40 MB en RAM por proceso; un worker basta y entra en el
  *free tier* (512 MB).

### 2.4. Contexto de build limpio (`.dockerignore`)

- Excluye lo pesado/irrelevante (`venv/`, cachés, `data/raw/`, frontend, tests…) para que el build
  suba un contexto chico y la imagen quede mínima.

## 3. Código puntual

| Qué | Dónde |
|-----|-------|
| Imagen de servicio (runtime, libgomp1, usuario, CMD) | `Dockerfile` |
| Subconjunto de runtime con pines exactos | `requirements-api.txt` |
| Contexto de build minimizado | `.dockerignore` |
| Diseño "carga, no entrena" | `docs/decisiones/0007-capa-api-fase3.md` |

## 4. Resultados (criterio de "hecho")

- ✅ **Dockerfile funcional** para la API: build → imagen autosuficiente con artefactos horneados.
- ✅ La app **se levanta en contenedor de forma reproducible** y hace *bind* a `0.0.0.0:$PORT`.
- ✅ Imagen mínima, capa de dependencias cacheable, usuario sin privilegios, `libgomp1` resuelto.
- 🟡 **`docker-compose` para API + base de datos:** pendiente; requiere primero el servicio de
  PostgreSQL (**EN-002**). Hoy la API se contieneriza sola (la persistencia corre en SQLite local).

```powershell
# Construir y correr localmente
docker build -t spc-api .
docker run --rm -p 8000:8000 spc-api
# Salud:  http://127.0.0.1:8000/health   ·   Swagger: http://127.0.0.1:8000/docs
```
