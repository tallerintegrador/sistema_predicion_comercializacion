# EN-002 — Configurar base de datos PostgreSQL

> **Habilitador técnico.** Persistencia de los datos de entrada y/o resultados del servicio.
> No entrega valor de usuario directo: habilita acumulación de corpus, históricos y auditoría.

- **Ámbito:** Infraestructura / persistencia. **Sprint 2: Semana 12.**
- **Estado:** 🟡 **Parcial.** Hoy la persistencia corre sobre **SQLite local** (ADR-0011);
  la migración a **PostgreSQL** queda **pendiente** (es la meta de este habilitador para producción).
- **Código clave:** `src/spc/service/repositorio.py`, `src/spc/service/corpus.py`,
  `src/spc/service/repositorio_auth.py`.
- **Decisión:** `docs/decisiones/0011-persistencia-corpus-incremental.md`.

---

## 1. Objetivo y alcance

Dotar al sistema de una **capa de persistencia** para:

- acumular el `history` que cada cliente envía (corpus incremental, deduplicado),
- guardar usuarios/credenciales del control de acceso por roles (ADR-0014),
- dejar el terreno listo para una base **multiusuario y concurrente** (PostgreSQL) en despliegue.

> **Nota de honestidad.** El criterio de aceptación de la tarjeta apunta a **PostgreSQL**
> (base accesible desde la API, esquema mínimo, conexión por variables de entorno). Lo que
> existe **hoy** es una persistencia equivalente en **SQLite** —misma capa de servicio, mismo
> contrato— pensada para correr **sin servidor de base de datos** durante Fase 3/4. PostgreSQL
> es la evolución natural de esta misma capa; abajo se documenta lo hecho y lo que falta.

## 2. Qué se hizo (estado actual: SQLite)

### 2.1. Capa de persistencia aislada (`service/repositorio.py`, `service/corpus.py`)

- Toda la lectura/escritura pasa por **`spc.service.repositorio`** (y `corpus.py`): la API y los
  servicios **no abren conexiones a mano**. Esto es lo que hace **portable** el cambio de motor de
  base de datos: migrar a PostgreSQL es reemplazar el backend de esta capa, no tocar la API.
- **Corpus incremental** (ADR-0011): cada petición que trae `history` lo **acumula deduplicado**
  por `(client_id, date, store_id, product_id)`. El modelo se sigue sirviendo **congelado**
  (ADR-0009); la base solo **acumula** para habilitar mejoras futuras.

### 2.2. Persistencia de identidad (`service/repositorio_auth.py`)

- Usuarios, roles y credenciales del control de acceso (ADR-0014) viven en la base: las cuentas de
  demostración se **siembran hasheadas** (scrypt) al arrancar; no hay contraseñas en claro.

### 2.3. Ubicación configurable por entorno

- La ruta/origen de la base se resuelve por **variable de entorno** (no hard-codeada), de modo que
  el mismo código apunte a un fichero local en desarrollo o a un servidor gestionado en despliegue.

| Qué | Dónde |

| Capa de persistencia (corpus + lectura/escritura) | `src/spc/service/repositorio.py` |
| Acumulación incremental del corpus | `src/spc/service/corpus.py` |
| Persistencia de usuarios/credenciales | `src/spc/service/repositorio_auth.py` |
| Punto de captura del corpus en el ruteo | `src/spc/api/ruteo.py` |

## 4. Resultados (criterio de "hecho")

- ✅ Existe una **capa de persistencia única** que aísla a la API del motor de base de datos.
- ✅ El **corpus** se acumula deduplicado por cliente (ADR-0011), sin tocar el modelo congelado.
- ✅ La **identidad** (usuarios/roles) persiste con credenciales hasheadas.
- ✅ El origen de la base se configura por **variable de entorno**.
- 🟡 **Pendiente para PostgreSQL:** proveedor gestionado (Render/Railway), `DATABASE_URL`,
  driver (`psycopg`), DDL del esquema mínimo y, en el contenedor, `docker-compose` con el servicio
  de base (ver **EN-003**) y prueba de conexión desde la API desplegada (ver **EN-004**).
