# TA-004 — Implementar arquitectura en capas

> **Tarea técnica.** Separar el sistema en tres capas con **dependencias hacia adentro** —API →
> Servicio → Motor— de modo que cada una tenga una única responsabilidad y el motor no conozca HTTP.

- **API:** `src/spc/api/` (`main.py`, `routers/`, `schemas/`, `dependencies.py`, `errors.py`).
- **Servicio:** `src/spc/service/` (`adaptador.py`, `artefactos.py`, `*_service.py`, `errores.py`).
- **Motor:** `src/spc/models/` (`Predictor*`, `Perfilador*`).
- **Decisión:** `docs/decisiones/0007-capa-api-fase3.md`.

---

## 1. Objetivo y alcance

```
HTTP / API  ──►  Servicio (negocio)  ──►  Motor de ML (artefactos)
 conoce HTTP       conoce el contrato        carga y predice;
 y el contrato     y las reglas de negocio   NO conoce HTTP
```

Regla central: **las dependencias apuntan hacia adentro**. El **motor no importa nada de la API**; el
**servicio no importa `spc.api`** (los routers convierten Pydantic↔dict; el servicio trabaja con
estructuras de Python/pandas). Así cada capa se prueba y evoluciona por separado.

## 2. Qué se hizo

### 2.1. Capa API (`src/spc/api/`)

- **`main.py`** — *app factory* `crear_app`: arma FastAPI, CORS, lifespan, routers, manejadores de
  error y Swagger. **Carga los artefactos una sola vez** en el lifespan y los guarda en
  `app.state.registro`. `crear_app` acepta un registro inyectado → los tests usan artefactos diminutos
  sin tocar disco.
- **`routers/`** — un `POST` por dominio (`/sales`, `/purchases`, `/inventory`); routers **finos**
  que validan, convierten y delegan en el servicio.
- **`schemas/`** — contrato Pydantic (TA-001).
- **`dependencies.py`** — `obtener_registro` (inyección del motor por `Depends`).
- **`errors.py`** — traduce excepciones al cuerpo de error del contrato (TA-005).

### 2.2. Capa Servicio (`src/spc/service/`)

- **`adaptador.py`** — única pieza que conoce **ambos** esquemas (contrato ↔ motor); traduce (TA-002).
- **`artefactos.py`** — carga y resuelve versión de los artefactos (TA-003).
- **`ventas_service` / `compras_service` / `almacen_service`** — la **lógica de negocio** por dominio
  (HU-001/002 reúso del pronóstico, HU-003/004 ensamblado clasificación+clustering).
- **`errores.py`** — excepciones de dominio **independientes de HTTP** (`SolicitudInvalida`).

### 2.3. Capa Motor (`src/spc/models/`)

Objetos `Predictor*`/`Perfilador*` autocontenidos: reciben/devuelven pandas, **no conocen HTTP ni el
negocio del cliente**. Exponen interfaces estables (`pronosticar_horizonte`, `predecir`, `perfilar`)
que el servicio llama sin conocer el algoritmo.

### 2.4. Inyección de dependencias y arranque

Artefactos cargados en el **lifespan** (no por petición) → los routers los reciben por
`Depends(obtener_registro)`. Si el motor no está cargado, `obtener_registro` lanza
`ServicioNoDisponible` (→ 503). CORS configurable por `SPC_CORS_ORIGINS`.

## 3. Código puntual

| Qué | Dónde |
|-----|-------|
| App factory + lifespan (carga única) | `src/spc/api/main.py:61` (`crear_app`) |
| Inyección del motor a los routers | `src/spc/api/dependencies.py:17` (`obtener_registro`) |
| Frontera contrato↔motor | `src/spc/service/adaptador.py` |
| Excepción de dominio (sin HTTP) | `src/spc/service/errores.py:13` (`SolicitudInvalida`) |
| Routers finos (un POST por dominio) | `src/spc/api/routers/{ventas,compras,almacen}.py` |
| ADR de la arquitectura | `docs/decisiones/0007-capa-api-fase3.md` §1 |

## 4. Resultados

- **Separación verificada:** el motor no importa la API; el servicio no importa `spc.api` (ADR 0007,
  criterio de hecho).
- **Intercambiabilidad:** cambiar de artefacto no toca código (glob de versión, TA-003).
- **Testabilidad:** `crear_app(registro=…)` permite construir la app con artefactos diminutos →
  `tests/api` corre **sin GPU ni `data/raw`**.
- **Una sola carga** del motor en el arranque (no por petición) → respuestas rápidas.
