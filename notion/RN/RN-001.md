# RN-001 — Separación estricta de capas

> **Requisito no funcional / restricción de arquitectura.** El sistema debe mantener tres capas con
> **dependencias en un solo sentido** (API → Servicio → Motor). Ninguna capa interior puede depender
> de una exterior.

- **Tipo:** Restricción arquitectónica (no funcional).
- **Ámbito:** todo el paquete `spc` (`api/`, `service/`, `models/`).
- **Evidencia:** `docs/decisiones/0007-capa-api-fase3.md` (criterio de hecho), `docs/plan_maestro_spc.md` §2.
- **Implementación documentada en:** TA-004.

---

## 1. Enunciado de la regla

```
HTTP / API  ──►  Servicio (negocio)  ──►  Motor de ML (artefactos)
 conoce HTTP       conoce el contrato        carga y predice;
 y el contrato     y las reglas de negocio   NO conoce HTTP
```

- El **motor de ML (`spc.models`) no importa nada de la API ni del servicio.**
- El **servicio (`spc.service`) no importa `spc.api`.**
- La API es la única capa que conoce HTTP/FastAPI; el motor es la única que conoce los algoritmos.

> Del plan maestro (§2): *"Modular por capas. Cada capa tiene una responsabilidad única y se puede
> testear y reemplazar de forma aislada."* RN-001 es esa modularidad **como regla verificable**.

## 2. Por qué importa (valor)

- **Reemplazabilidad:** se puede cambiar el motor (otro artefacto, otro algoritmo) sin tocar la API,
  y exponer otra interfaz (CLI, batch) sin tocar el motor.
- **Testabilidad:** cada capa se prueba aislada (el motor con datos sintéticos; la API con artefactos
  diminutos inyectados).
- **Reutilización del motor offline:** entrenamiento y scripts usan `spc.models` **sin arrastrar
  FastAPI**.
- **Claridad de responsabilidades:** un error de validación es de la API; una regla de negocio es del
  servicio; una métrica es del motor.

## 3. Cómo se cumple (mecanismo)

- **Dirección de imports:** `api → service → motor`. Los routers convierten Pydantic↔dict; el servicio
  trabaja con estructuras de Python/pandas; el motor recibe/devuelve pandas.
- **Frontera explícita:** `service/adaptador.py` es la **única** pieza que conoce ambos esquemas
  (contrato y motor).
- **Errores desacoplados de HTTP:** `service/errores.py` define `SolicitudInvalida` (subclase de
  `ValueError`) **sin conocer FastAPI**; la API la traduce a HTTP (RN/TA-005).
- **Inyección de dependencias:** los routers reciben el motor por `Depends(obtener_registro)`; no lo
  importan ni lo cargan.

## 4. Cómo se verifica (prueba)

1. **Chequeo de imports (evidencia objetiva).** No hay ningún import de la API en el servicio, ni de
   la API/servicio en el motor:

   ```bash
   # service/ NO importa spc.api  →  0 coincidencias
   grep -rE "^\s*(from|import)\s+spc\.api" src/spc/service
   # models/ NO importa spc.api ni spc.service  →  0 coincidencias
   grep -rE "^\s*(from|import)\s+spc\.(api|service)" src/spc/models
   ```

   Ambos comandos devuelven **0 coincidencias** → la regla se sostiene a nivel de código.

2. **Criterio de hecho del ADR 0007:** casillas verificadas *"el motor de ML no importa nada de la
   API; el servicio no importa la API"*.

3. **Tests:** `tests/api/conftest.py` construye la app con `crear_app(registro=…)` inyectando
   artefactos diminutos → ejercita la API **sin** que el motor conozca la API y **sin GPU/`data/raw`**.

## 5. Resultado

- Regla **cumplida y verificable** con un grep (parte del criterio de hecho de Fase 3).
- La API sobrevive a un cambio de artefacto sin tocar código (glob de versión, TA-003); el motor se
  entrena/usa offline sin dependencias web.
